Splunking DLP data with a side of active response

devs  included in  category Blue Team
 2022-01-19  2022-01-19  592 words   3 minutes 

Recently, I was attending a corporate event via conference call and I noticed a distinct phrase being mentioned over and over by the event speakers. “Please refrain from taking screenshots of these slides as they are not meant for public disclosure.” Sound familiar to anyone? How many people are actually going to abide by this rule?

For in-person events, this warning may be enough to ward off the rogue mobile phone picture taker…but let’s face it…when people are at home, in their office, not thinking that they are being “watched”, they definitely are going to take that screen capture (and potentially send it elsewhere). Don’t believe me?

The following sections outline how I tested this theory and built out a very rough active response workflow for preventing data loss scenarios of this type.

System Level Log Monitoring

I should mention before going further that although the latest wiz-bangy EDR tool isn’t needed for this setup, there needs to be some method, tool, or mechanism of grabbing system level file creation events from the to-be monitored devices (preferably quickly).

For those unfamiliar, there are some great free tools out there that can be deployed for this purpose. They include Velociraptor and everyone’s favorite, Sysmon . Additionally, for macOS systems, the product Jamf Protect works very well for remotely collecting Apple Unified Log events.

Most organizations these days are sending various categories of event logs to a SIEM (i.e. Splunk) and endpoint data should be no different. Features such as Splunk HEC and Universal Forwarders make it easy to pull in this information.

So what do we look for in the event logs? In our example scenario of a meeting participant taking an “unauthorized” screen capture of sensitive content, it’s pretty easy. Any file creation events that occur from the devices default screen capture application are likely to be files of interest.

Jamf Protect even comes with an out-of-box analytic for screenshots being taken on a system. https://docs.jamf.com/jamf-protect/documentation/Security.html

But what about individuals that are using non-native, third-party screen capture tools? I would recommend performing an inventory of software applications that are installed on corporate systems and performing some further research / testing for how these applications generate image files. Adjust Sysmon, Velociraptor, etc. queries as needed once the general process hierarchy pattern is identified.

Now that the event has been logged, a SHA256 hash should be included within the event. We can utilize Splunk’s search scheduling to regularly populate a lookup file with any and all file hashes that are observed from the screenshot events. But then what?

Active Response Actions

Splunk alert actions are great ways of integrating Splunk with other systems and Security, Orchestration, Automation, and Response (SOAR) platforms. With regards to this situation, one configuration that could be put in place is that, on a regular interval, Splunk can send the file hashes included in the previously created lookup file to a SOAR platform. The SOAR platform can interface with the organization’s DLP product, email platform, etc. to block the specified file hashes from being transferred to external locations. Despite email controls, I continue to see email being a common mechanism for data exfiltration by insider threat actors.

I’m no expert when it comes to data loss prevention (DLP) and there are definitely far better (and expensive product) methods for performing what I outlined today. Hell enforcing stricter system controls around screenshot captures and copy & paste are likely more effective. However, I realize going those routes may not work for everyone. This post is my attempt at providing a creative alternative solution.
